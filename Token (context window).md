ယခင် “chunking” အကြောင်းရှင်းပြရာတွင် “စက်ဘာသာပြန် Model အတွက် အဝင်အရှည်ကန့်သတ်ချက်ကို ကျော်လွှားရန်” ဟူသောအချက်ကို ထည့်သွင်းပြောကြားခဲ့ပါသည်။ ထိုကန့်သတ်ချက်ကို **context window** (သို့မဟုတ် **token limit**) ဟုခေါ်ပြီး၊ ယခုအခါ AI ဘာသာပြန်စနစ်များ၏ အဓိကကျသော နည်းပညာဆိုင်ရာ ဘောင်တစ်ခုဖြစ်ပါသည်။ အောက်တွင် အသေးစိတ် ရှင်းပြပေးပါမည်။

---

### ၁။ Token ဆိုသည်မှာ အဘယ်နည်း။
**Token** ဆိုသည်မှာ ဘာသာစကားမော်ဒယ် (Language Model) တစ်ခုက စာသားကို နားလည်ရန် ခွဲခြမ်းလိုက်သော **အသေးငယ်ဆုံးသော အခြေခံယူနစ်** ဖြစ်သည်။  

- စာလုံးတစ်လုံးချင်းစီသည် token တစ်ခုဖြစ်နိုင်သလို၊ စာလုံးတစ်ပိုင်းချင်း (sub-word) လည်း ဖြစ်နိုင်သည်။  
- ဥပမာ “မြန်မာ” ဟူသော စကားလုံးကို “မြန်” + “မာ” ဟု ခွဲနိုင်သလို၊ “မြ” + “န်” + “မာ” စသည်ဖြင့် ခွဲခြင်းမှာ အသုံးပြုသည့် tokenizer ပေါ်မူတည်သည်။  
- အင်္ဂလိပ်စာတွင် “transformer” ကို “transform” + “er” ဟု ခွဲတတ်ပြီး၊ နေရာလွတ် (space) နှင့် ပုဒ်ဖြတ်သင်္ကေတများသည်လည်း token များအဖြစ် ရေတွက်ခံရတတ်သည်။

**အရေးကြီးချက်** – Token သည် စာလုံးရေ (word count) နှင့် တိုက်ရိုက်မတူညီပါ။ မြန်မာလိုကဲ့သို့ ဝဏ္ဏအခြေပြု ဘာသာစကားများတွင် စာလုံးတစ်လုံးလျှင် token အများအပြားဖြစ်တတ်သောကြောင့် တူညီသော အဓိပ္ပာယ်ကို ဖော်ပြရန် အင်္ဂလိပ်ထက် token ပိုမိုသုံးစွဲရလေ့ရှိသည်။

---

### ၂။ Context Window (သို့) Maximum Sequence Length ဆိုသည်မှာ
**Context Window** ဆိုသည်မှာ AI မော်ဒယ်တစ်ခုက **တစ်ကြိမ်တည်းတွင် လက်ခံနိုင်သော (input) နှင့် ထုတ်လုပ်နိုင်သော (output) token စုစုပေါင်း၏ အများဆုံးပမာဏ** ဖြစ်သည်။  
အချို့မော်ဒယ်များတွင် input သီးသန့်ကန့်သတ်ချက်၊ output သီးသန့်ကန့်သတ်ချက်ဟူ၍ ခွဲခြားထားသော်လည်း အများအားဖြင့် context window ဆိုသည်မှာ input token များကို သိုမှီးနိုင်သည့် ပမာဏကို အဓိကရည်ညွှန်းသည်။

- ဥပမာ - GPT-3.5-turbo ၏ context window သည် 4,096 tokens (နောက်ပိုင်း 16,385 tokens) ဖြစ်သည်။  
- GPT-4-turbo သည် 128,000 tokens အထိ လက်ခံနိုင်သည်။  
- Google Gemini 1.5 Pro သည် 1 million tokens အထိ လက်ခံနိုင်စွမ်းရှိသည်။

ဤကန့်သတ်ချက်အတွင်း၌သာ မော်ဒယ်သည် အကြောင်းအရာကို နားလည်နိုင်၊ ဘာသာပြန်နိုင်၊ ဆက်စပ်မှုကို ထိန်းသိမ်းနိုင်သည်။

---

### ၃။ အဘယ့်ကြောင့် ဤကန့်သတ်ချက် ရှိရသနည်း။
AI မော်ဒယ်များ အထူးသဖြင့် **Transformer** ဗိသုကာကို အခြေခံထားသော မော်ဒယ်များသည် “self-attention” ယန္တရားကို အသုံးပြုပါသည်။ ဤယန္တရား၏ အဓိကအားနည်းချက်မှာ **quadratic complexity** ဖြစ်သည်။

- စာကြောင်းအတွင်းရှိ token တိုင်းသည် အခြားသော token တိုင်းနှင့် ဆက်စပ်မှုကို တွက်ချက်ရသည်။  
- token အရေအတွက် $n$ ရှိပါက တွက်ချက်မှုပမာဏသည် $n^2$ နှင့် အချိုးကျပါသည်။  
- $n$ နှစ်ဆဖြစ်သွားလျှင် memory သုံးစွဲမှုနှင့် အချိန်သည် လေးဆခန့် တိုးလာသည်။

ထို့ကြောင့် လက်တွေ့ကျသော GPU memory နှင့် တုံ့ပြန်ချိန် (latency) ကို စီမံနိုင်ရန် **ကန့်သတ်ချက်တစ်ခု** သတ်မှတ်ထားခြင်းဖြစ်သည်။ ခေတ်သစ်မော်ဒယ်များတွင် flash attention, sparse attention စသည့်နည်းပညာများဖြင့် window ကိုချဲ့ထွင်လာသော်လည်း အကန့်အသတ်မဲ့ မဟုတ်သေးပါ။

---

### ၄။ Context Window သည် ဘာသာပြန်ခြင်းအပေါ် မည်သို့သက်ရောက်မှုရှိသနည်း။
ဘာသာပြန်ရာတွင် input စာသားတစ်ခုလုံးနှင့် output အားလုံးသည် context window အတွင်း၌ ရှိရပါမည်။  
**ဥပမာ**  
- GPT-3.5 ကို သုံး၍ စာမျက်နှာ ၂၀ ရှည်သော စာတမ်းကို တစ်ခါတည်းဘာသာပြန်ခိုင်းလျှင် input token 4,096 ကျော်သွားပါက မော်ဒယ်က လက်ခံနိုင်မည်မဟုတ်ပါ။  
- ထို့ကြောင့် စာကို **chunk** များခွဲ၍ တစ်ခုချင်းစီ ဘာသာပြန်ရသည်။

**Chunking နှင့် Context Window ဆက်စပ်ပုံ**  
- Chunk တစ်ခု၏ token အရေအတွက်ကို မော်ဒယ်၏ window အတွင်းရှိစေရန် သတ်မှတ်ရသည်။  
- ဥပမာ 4,096 token window ရှိသော မော်ဒယ်အတွက် chunk တစ်ခုလျှင် မြန်မာစာ token ၃၀၀၀ ခန့်ထားပြီး prompt (ညွှန်ကြားချက်) အတွက် token ၅၀၀၊ output အတွက် နေရာ ၅၀၀ စသည်ဖြင့် ချန်ထားရသည်။  
- ဝါကျတစ်ကြောင်းတည်းကိုပင် chunk တစ်ခုအနေဖြင့် ပို့ပါက window ၏ အလွန်သေးငယ်သော အစိတ်အပိုင်းကိုသာ သုံးမိပြီး ရှေ့နောက်ဆက်စပ်မှု (context) ဆုံးရှုံးနိုင်သည်။ ထို့ကြောင့် **တတ်နိုင်သမျှကြီးမားသော chunk ကို window အတွင်းထည့်ပြီး overlap ပါထည့်ပေးခြင်း** သည် ပိုမိုကောင်းမွန်သော ဘာသာပြန်ကိုရရှိစေသည်။

---

### ၅။ Context Window ကန့်သတ်ချက်ကို ကျော်လွှားရန် နည်းဗျူဟာများ
1. **Sliding Window with Overlap** – Chunk တစ်ခုနှင့်တစ်ခုကြား token ၁၀၀ မှ ၅၀၀ ခန့် ထပ်နေအောင်ခွဲခြင်းဖြင့် စာပိုဒ်အကူးတွင် အဓိပ္ပာယ်ပျောက်ဆုံးမှုကို ကာကွယ်သည်။  
2. **Summarization-based Chunking** – ရှေ့ chunk ၏ အကျဉ်းချုပ်ကို နောက် chunk ၏ prompt တွင် ထည့်ပေးခြင်းဖြင့် context ကိုဆက်ထိန်းထားနိုင်သည် (သို့သော် ဘာသာပြန်တွင် အနည်းငယ် အချက်အလက်လွဲနိုင်သည်)။  
3. **Long-context Model သုံးခြင်း** – GPT-4 128k, Claude 200k, Gemini 1M ကဲ့သို့ window ကြီးမားသော မော်ဒယ်များသို့ ပြောင်းလဲအသုံးပြုခြင်းဖြင့် chunking လိုအပ်ချက်ကို လျှော့ချနိုင်သည်။ သို့သော် ကုန်ကျစရိတ်နှင့် တုံ့ပြန်ချိန် မြင့်တက်တတ်သည်။  
4. **Document-level Translation Models** – အချို့သော သီးသန့်ဘာသာပြန်မော်ဒယ်များ (e.g., Google Cloud Translation NMT, DeepL) သည် စာရွက်စာတမ်းတစ်ခုလုံးကို နောက်ကွယ်တွင် intelligent splitting ပြုလုပ်ပြီး context caching ဖြင့် ဘာသာပြန်ပေးနိုင်သောကြောင့် user က chunking ကို စိုးရိမ်စရာမလိုပေ။

---

### ၆။ မြန်မာဘာသာအတွက် အထူးဂရုပြုရမည့်အချက်
မြန်မာစာသည် သရအက္ခရာ၊ ဗျည်းအက္ခရာများကို ပေါင်းစပ်ရေးသားသော အချပ်ရေးစနစ်ဖြစ်ပြီး စာလုံးတစ်လုံးချင်းကြား နေရာလွတ်မရှိသလောက်နည်းပါသည်။ ထို့ကြောင့် tokenization ပြုလုပ်ရာတွင်:

- စာလုံးတစ်လုံးကို syllable 단위로 ခွဲခြင်းကြောင့် စာလုံးရေအနည်းငယ်ဖြင့် token အမြောက်အများ ဖြစ်ပေါ်တတ်သည်။  
- ဥပမာ “ကျွန်တော်တို့သည်” ဟူသော စာတန်းတိုလေးသည် အင်္ဂလိပ်လို “We” 1 token ဖြစ်နိုင်သော်လည်း မြန်မာလို token ၅-၆ ခုထိ ဖြစ်နိုင်သည် (tokenizer အပေါ်မူတည်)။  
- ထို့ကြောင့် **မြန်မာစာကို ဘာသာပြန်ရာတွင် အင်္ဂလိပ်စာထက် context window ကို ပိုမိုလွယ်ကူစွာ ပြည့်သွားစေနိုင်သည်**။  
- Chunk size သတ်မှတ်ရာတွင် မြန်မာစာ token အရေအတွက်ကို အမြဲစစ်ဆေးပြီး သင့်လျော်အောင်ချိန်ညှိသင့်သည် (ဥပမာ - tiktoken, sentencepiece library များ သုံးတွက်ခြင်း)။

---

### နိဂုံးချုပ်
**Context Window** သည် စက်ဘာသာပြန်မော်ဒယ်တစ်ခုက တစ်ပြိုင်နက်တည်း စဉ်းစားနားလည်နိုင်သော စာသားပမာဏ၏ နည်းပညာဆိုင်ရာ ကန့်သတ်ချက်ဖြစ်ပါသည်။ ၎င်းကြောင့် ရှည်လျားသောစာများကို ဘာသာပြန်ရန် **chunking** သည် မရှိမဖြစ်လိုအပ်လာပြီး၊ အကန့်အသတ်ကိုသိရှိနားလည်မှသာ အရည်အသွေးမြင့်၊ အဓိပ္ပာယ်ဆက်စပ်မှုမပျက်သော ဘာသာပြန်ရလဒ်ကို ရရှိနိုင်မည်ဖြစ်သည်။
